세 프로젝트에서 Claude Code를
어떻게 구성하고 함께 쓸까

우리는 한 작업이 여러 프로젝트(프론트·백엔드·RAG)를 건드리는 일이 잦습니다. 그런데 Claude 기록이 지금 열린 한 곳에만 남고, git에 공유도 안 되고 있습니다. 이걸 어떻게 정리할지 처음부터 차근차근 봅니다.

📌 30초 요약
  • Claude 구성은 "팀과 나눌 것" 과 "내 PC에만 둘 것" 으로 나뉜다.
  • 나눌 것(프로젝트 설명·명령·작업 기록)은 git에 커밋해서 공유한다.
  • 한 작업이 여러 프로젝트를 건드리면 각 프로젝트에 기록을 남긴다.
  • 모듈이 많아도 작업 기록은 프로젝트당 한 세트, 모듈은 태그로 구분.

1 지금 우리 상황

우리 코드는 세 개의 독립된 프로젝트로 나뉘어 있고, 여러 명이 이들을 오가며 개발합니다.

🎨 프론트
sfa-client-main
⚙️ 백엔드
sfa-server · 모듈 多
🔎 RAG
sfa-rag

한 기능이 백엔드+프론트를 같이 고치기도 하고, RAG만 고치기도 합니다. 각자 Claude 구성을 만들어 쓰지만 자기 PC에만 있고 git에 안 올려서 서로 못 봅니다. 그래서 풀 문제가 셋입니다.

1
공유가 안 됨 — 기록을 git에 안 올려서 동료도 나도 서로의 기록을 못 본다.
2
크로스-프로젝트 기록이 한 곳에만 — RAG에서 작업하며 백엔드도 고쳤는데 기록은 열린 RAG에만 남는다.
3
모듈이 많을 때 문서 구성 — 백엔드·프론트는 모듈이 많아 docs/library/처럼 폴더로 나눌지 고민.

2 공유할 것 vs 내 PC에만 둘 것

Claude 구성은 네 종류입니다. 셋은 공유하고, 하나는 개인용으로 둡니다.

종류무엇인가 / 예시어떻게
① 프로젝트 설명이 프로젝트의 기술·DB·규칙 — CLAUDE.md🟢 커밋(공유)
② 작업 명령시작/종료 명령·권한 — .claude/skills, .claude/settings.json🟢 커밋(공유)
③ 작업 기록진행 로그·결정 — docs/PROGRESS.md 등🟢 커밋(공유)
④ 개인 구성내 PC 전용 메모·권한 — settings.local.json, CLAUDE.local.md🔴 커밋 안 함(개인)
비유 — ①②③은 "팀 위키"(모두 보고 함께 갱신), ④는 "내 책상 포스트잇"(나만 본다).

경계는 .gitignore가 지킨다

각 프로젝트에 이 줄을 넣으면 ④(개인)는 자동으로 무시되고 ①②③만 커밋됩니다.

# Claude 개인 구성 — 커밋하지 않음
.claude/settings.local.json
CLAUDE.local.md

3 각 프로젝트에 구성 심기

지금은 구성이 로컬에만 있으니, 세 프로젝트 각각에서 한 번씩 합니다.

  1. 이미 있는 Claude 구성(CLAUDE.md·.claude·docs)을 확인
  2. 위 .gitignore를 추가 (개인 파일 제외)
  3. 공유 대상 파일을 git에 커밋·푸시
# 각 프로젝트에서 한 번씩 (sfa-client-main, sfa-server, sfa-rag)
git add CLAUDE.md .claude docs .gitignore
git commit -m "chore: Claude Code 팀 공유 구성 추가"
git push
✅
결과 — 동료가 git pull 하면 Claude 구성도 따라옵니다. 누가 어디서 열든 같은 규칙·같은 기록을 공유 → 1번 문제 해결.
⚠️
진짜 비밀번호·키는 절대 커밋하지 않습니다. (별도 위치 _secure_config 등에 둡니다.)

2번 문제 크로스-프로젝트 기록 풀기

한 작업이 여러 프로젝트를 건드릴 때, 기록이 한 곳에만 남는 문제입니다.

왜 한 곳에만 기록될까? (쉬운 풀이)

Claude는 지금 열린 폴더의 기록만 읽고 씁니다. 종료 기록 명령(/wrap)도 그 폴더의 docs/에만 씁니다. 그래서 RAG에서 열어 백엔드까지 고쳐도 기록은 RAG에만 남고, 백엔드 동료는 맥락을 못 봅니다.

비유 — 두 부서가 함께 한 회의인데 회의록을 한 부서 폴더에만 저장한 셈. 다른 부서는 그 결정이 있었는지 모릅니다.
🎯
해결 원칙 — 작업이 건드린 프로젝트마다 기록을 남긴다. 자세한 본문은 주로 작업한 프로젝트 한 곳, 다른 프로젝트엔 짧은 한 줄 + 가리키는 링크.
예시 — RAG에서 작업했고 백엔드도 고쳤다면:
sfa-rag/docs/PROGRESS.md자세한 본문 기록
sfa-server/docs/PROGRESS.md"RAG 작업으로 함께 변경됨 → sfa-rag PROGRESS 참고" 한 줄

→ 어느 프로젝트 동료든 자기 프로젝트 기록만 봐도 "왜 바뀌었는지" 따라갈 수 있습니다.

어디까지 공유할지는 팀이 정한다 — 3가지 방법

방법 A
각 프로젝트 기록 + 교차 한 줄추천
방법 B
공유 기록 저장소(서브모듈)
방법 C
하이브리드 (결정만 공유)

위 원칙 그대로. 새로 만들 저장소가 없습니다.

👍 장점

  • 가장 단순 — 새 인프라 0
  • 팀원은 자기 프로젝트만 보면 됨

👎 단점

  • 여러 프로젝트 걸친 결정은 살짝 중복
  • 전체 흐름은 한눈에 안 보임

크로스-프로젝트 기록만 모으는 공용 저장소(서브모듈)를 따로 둡니다.

👍 장점

  • 기록이 한 곳 — 분기 없음
  • 전체 흐름을 다 봄

👎 단점

  • 새 저장소 + 서브모듈 운영 부담
  • 전 팀이 추가 git 습관 필요

중요한 "결정"만 공용, 일상 "진행 로그"는 프로젝트별.

👍 장점

  • 중요 결정은 안 엇갈림
  • 일상 로그는 가볍게

👎 단점

  • 공용 저장소 1개는 운영
  • "이건 결정? 로그?" 판단 필요

자동화 팁 — 자주 함께 고치는 프로젝트를 각 .claude/settings.json의 additionalDirectories에 넣어두면(예: 프론트에 ["../sfa-server"]), Claude가 그 프로젝트까지 접근해 교차 기록을 자동으로 남깁니다. 방법 A의 "각 프로젝트에 기록"이 사람 손 없이 됩니다.

3번 문제 모듈이 많을 때 docs 구성

백엔드(10모듈)·프론트(여러 앱)처럼 모듈이 많아도 규칙은 단순합니다.

작업 기록 3종 — 프로젝트당 "한 세트" (모듈로 안 쪼갬)

PROGRESS·DECISIONS·PROJECT_PLAN은 모듈로 나누지 않고, 항목 앞에 모듈 태그를 붙입니다.

- [library] 자료 카드 정렬 버그 수정
- [work-support] 게시판 카테고리 트리 추가

왜? ① /resume·/wrap이 정해진 한 파일을 읽고 씀(쪼개면 깨짐) ② 한 작업이 여러 모듈을 건드려 시간 흐름이 흩어짐. PROJECT_PLAN은 한 파일 안 모듈별 섹션으로 충분.

주제·참고 문서 — 평면 시작, 쌓이면 폴더 승격

모듈 문서 1~2개평면 docs/library-카드정렬.md (이름 접두)
모듈 문서 3개 이상docs/library/ 폴더로 승격
빈 모듈 폴더 미리 만들기하지 않음
어느 경우든docs/README.md 색인 유지
예시 — 한 프로젝트의 docs/ 내부
docs/
├── PROGRESS.md          # 한 세트 — 모듈은 [library] 태그
├── DECISIONS.md         # 한 세트
├── PROJECT_PLAN.md      # 한 파일 안에 모듈별 섹션
├── README.md            # 어디에 뭐가 있는지 색인
├── library-카드정렬.md      # 적을 땐 평면 + 이름 접두
└── work-support/            # 3개 이상 쌓이면 폴더 승격
    ├── 게시판-규칙.md
    ├── 카테고리-트리.md
    └── 워크플로-명세.md
💡
한마디로 — 기록은 한 세트(+태그), 주제 문서는 적으면 평면·많으면 폴더. 미리 폴더부터 파지 말 것.

6 매일 어떻게 쓰나

  1. 프로젝트 폴더에서 Claude 실행
  2. /resume → 지난 기록 읽고 "지난 작업 X, 다음 Y?" 확인
  3. 작업 → 코드 수정
  4. /wrap → 오늘 한 일을 docs/PROGRESS.md에 기록
  5. git 커밋·푸시 (변경 기록도 함께 공유)
🔀
여러 프로젝트를 고쳤다면 — 주로 작업한 프로젝트에 자세히 기록·커밋하고, 함께 바뀐 프로젝트에도 한 줄 교차 기록·커밋. 커밋은 프로젝트마다 따로(각각 독립 git).
