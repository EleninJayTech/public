cross-repo + 모노레포 동시성을
하나로 합친 문서 구조

두 조사를 합쳤습니다. 경쟁이 아니라 다른 층을 다루고 있었고, 충돌점 하나(작업 기록 분할)는 동시성 기준으로 정리했습니다.

(A) cross-repo 조사 — 3 repo 사이 조정 (B) 프론트팀 논의서 — 모노레포 내부 동시성
📌 한 장 요약
  • 두 조사는 다른 층 (A) repo 간 / (B) 한 repo 안 → 둘 다 채택, 층으로 결합.
  • 작업 기록 분할은 동시 편집자 수로 결정 — 저동시성=1세트+[모듈]태그, 고동시성 모노레포=앱별 폴더 분할.
  • 토큰은 구조로 거의 안 줄어든다. 분할의 진짜 가치는 Git 충돌 차단 + 타겟 정확도.
  • 어느 층이든 "여러 대상을 건드리면 각 대상에 기록(방법 A)"은 동일.

1 두 층 모델 (핵심 프레임)

두 문서는 서로 다른 층을 다룹니다. 독립적으로 동시에 적용됩니다.

inter-repo · (A) 우리 조사

단위: repo 3개(프론트·백엔드·RAG) · 문제: 한 작업이 여러 repo를 건드림 · 해법: additionalDirectories · 방법 A · repomix

▲ 이 좌표계 위에 ▼ 고동시성 repo(프론트)는 아래 층을 얹는다
intra-repo · (B) 프론트팀

단위: 한 repo 안 앱/모듈 · 문제: 여러 명이 한 PROGRESS.md에 append → Git 충돌 · 해법: 앱별 분할

2 작업 기록 분할 — 동시성 기준 결정

(A)는 "분할 금지(1세트+태그)", (B)는 "솔루션별 분할". 정답은 동시 편집자 수에 달려 있습니다.

상황권장이유
한 repo에 개발자 1~2명, 모듈 경계 흐림1세트 + [모듈] 태그타임라인 보존·단순, 충돌 드묾
한 repo에 여러 명·앱별 담당 고정앱별 폴더 분할(③안)Git 충돌 원천 차단 > 타임라인 이점

3 repo 적용

  • sfa-client-main (Turborepo, 6~7앱·앱별 담당) → 앱별 폴더 분할
  • sfa-server / sfa-rag (동시성 낮음) → 1세트 + [모듈] 태그 (편집 인원 늘면 분할로 전환)
💡
분할은 충돌 잦은 PROGRESS/DECISIONS만. 저빈도 마스터 PROJECT_PLAN은 루트에 공유(전사 인덱스). — (B) ③안의 영리한 절충.

3 토큰의 진실 ((B) 분석 반영)

  • Claude Code는 docs/ 문서를 자동으로 선로딩하지 않음 — 도구로 필요할 때만 읽음. → "구조로 토큰 수십% 절감"은 사실이 아님.
  • 오히려 앱 디렉토리의 CLAUDE.md는 그 폴더 작업 시 lazy-load되어 정적 컨텍스트가 소폭 늘 수 있음.
  • 결론: 구조는 토큰이 아니라 ① Git 충돌 차단 ② 타겟 정확도(남의 문서 오작성 방지) ③ 가독성 기준으로 고른다.

4 프론트 모노레포 구조 — ③안 채택

sfa-client-main/
sfa-client-main/
├── CLAUDE.md                  # [전역] 프론트 공통 규칙 + 문서·기록 규칙 블록
├── .gitignore                 # .claude/settings.local.json, CLAUDE.local.md 제외
└── docs/
    ├── PROJECT_PLAN.md        # [공통] 마스터 로드맵 + 활성 앱 인덱스 (루트 공유)
    ├── README.md              # 색인
    ├── work-support/          # 📂 앱별 폴더 (충돌 0)
    │   ├── PROGRESS.md        #   진척도 (이 앱 담당자만 append)
    │   ├── DECISIONS.md
    │   └── 게시판-규칙.md       #   [특화] 주제 문서
    └── approval/
        ├── PROGRESS.md
        └── 전자결제-라인-컨벤션.md

앱별 폴더라 서로 다른 파일을 편집 → Git 충돌 사라짐. 루트 PROJECT_PLAN.md만 공유(가끔 충돌하나 저빈도·단순 머지).

5 /resume·/wrap 을 솔루션-aware 로 (분할이 동작하는 핵심)

기록을 docs/<app>/로 쪼개면 skill이 어느 앱 문서를 읽고 쓸지 판별해야 합니다.

# /resume (모노레포·솔루션-aware)
대상 앱 판별 우선순위:
1) 현재 작업 디렉토리(PWD)가 apps/<app> 안이면 → 그 <app>
2) 아니면 현재 브랜치명에서 추출(feature/work-support → work-support)
3) 둘 다 불명확하면 → "오늘 작업할 앱을 골라주세요: [main, work-support, approval...]" 질문
판별 후: docs/<app>/PROGRESS.md 최상단 + docs/PROJECT_PLAN.md를 읽고
"이 앱의 지난 작업 X, 다음 Y?"로 보고.

# /wrap (모노레포·솔루션-aware)
- 같은 우선순위로 대상 앱 판별 → docs/<app>/PROGRESS.md 최상단에 [상태] 포함 기록.
- 완료 모호 시 독단 기록 금지, "남은 로직 있는데 [⏳미완]으로 추가할까요?" 확인.
↔️
sfa-server/sfa-rag(미분할)는 기존 단순 skill 그대로(고정 docs/PROGRESS.md).

6 여러 대상 동시 작업 = 방법 A (두 층 공통)

분할은 "어디에 기록?" 문제를 앱 층에서 재현합니다(앱 A+B 동시 수정). repo 층과 같은 원칙으로 해결.

🎯
주로 작업한 대상에 본문, 함께 바뀐 대상엔 [공통] 교차 한 줄 + 링크.
repo 간다른 repo가 additionalDirectories로 접근 가능하면 자동 교차기록, 아니면 "기록 필요" 알림
앱 간다른 앱 docs/<app>/PROGRESS.md에 [공통] … → work-support PROGRESS 참고 한 줄

7 상태 taxonomy (선택)

우리 4섹션(작업/결정/다음/미해결)은 유지하되, 항목에 상태 뱃지를 병기:

🚀 Done 완결 — PR/병합 가능한 독립 완결
⏳ Pending 미완 — 정상 빌드, 잔여 스펙 있어 이어서
🚨 Blocked 블록 — 컴파일 에러/외부 의존 교착으로 동결
- [work-support][Done] 게시판 카테고리 트리 추가
- [approval][Blocked] 결제선 자동지정 — 백엔드 API 미개발 대기

8 통합 CLAUDE.md 규칙 블록 (복붙)

각 repo CLAUDE.md 끝에 추가. 이 블록이 아래 규칙을 Claude가 자동 적용하게 합니다.

## 문서·기록 규칙 (Claude가 자동 적용)

### 작업 기록 구조 (동시성 기준)
- 저동시성 repo: PROGRESS·DECISIONS·PROJECT_PLAN을 한 세트로 두고 항목 앞에 [모듈] 태그.
- 고동시성 모노레포(이 repo가 해당이면): PROGRESS·DECISIONS는 docs/<app>/ 로 분리,
  마스터 PROJECT_PLAN.md만 docs/ 루트에 공유. /resume·/wrap은 대상 앱을 PWD>브랜치>질문 순으로 판별.
- 기록 항목엔 상태 [Done]/[Pending]/[Blocked]를 병기한다.

### 여러 대상 동시 변경 (방법 A)
- 주로 작업한 대상(repo 또는 app) docs에 본문 기록, 함께 바뀐 대상엔 [공통] 교차 한 줄 + 링크.
- 다른 repo는 접근 가능(cwd 또는 .claude/settings.json의 additionalDirectories)하면 자동 기록,
  불가하면 사용자에게 "○○에도 기록 필요"를 알린다.

### 주제·참고 문서 배치
- 평면(docs/*.md, 이름 접두)으로 시작 → 한 모듈/앱 주제 문서 3개+면 docs/<모듈>/ 승격. 빈 폴더 금지. README 색인 유지.

### 길이 관리
- PROGRESS가 약 800줄/분기 경계를 넘으면 가장 오래된 분기를 docs/archive/PROGRESS-<기간>.md로 옮기고
  활성 파일 맨 아래 포인터 한 줄. DECISIONS는 [[DEC-NNN]] 참조 보호 위해 상단 색인 유지 후 본문만 이동.

### 공유 vs 개인 / 시크릿
- 공유(커밋): CLAUDE.md·.claude/skills·.claude/settings.json·docs/.
- 개인(커밋 금지): .claude/settings.local.json·CLAUDE.local.md.
- 시크릿(.env·키·비밀번호)은 읽지도 커밋하지도 않는다.

### 토큰 참고
- docs/는 자동 선로딩되지 않고 필요 시만 읽힌다. 구조는 토큰이 아니라 충돌 차단·타겟 정확도 기준으로 고른다.
